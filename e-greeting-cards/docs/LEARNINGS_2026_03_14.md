## Bugs Fixed & Why They Happened

### 1. Docker Build Failing ŌĆö `sharp` Package

**What broke**: `npm install` inside Docker failed with errors from `@img/sharp-*` native packages. These are platform-specific binaries that have known broken registry entries.

**Why it happened**: `sharp` (used by Next.js for image optimisation) tries to install platform-native binaries as optional dependencies. On certain environments (WSL2 + Docker), the resolution fails and the entire `npm install` crashes.

**Fix**:
```json
// package.json
"overrides": { "sharp": "0.32.6" }
```
```dockerfile
# Dockerfile
RUN npm install --omit=optional
```
`--omit=optional` skips the broken native packages entirely. The app works fine without them in development.

**Lesson**: Always check if failing packages are `optionalDependencies`. Using `--omit=optional` is safe for dev and staging; revisit for production image optimisation at scale.

---

### 2. Docker Build Failing ŌĆö Lock File Version Mismatch

**What broke**: `npm ci` inside Docker failed because `package-lock.json` was generated by npm@11 (local machine) but Docker runs npm@10. The lock file format differs between versions.

**Why it happened**: The `package-lock.json` was committed and Docker tried to use `npm ci` (strict lock file mode), which rejects a lock file from a different npm version.

**Fix**:
```dockerignore
# .dockerignore
package-lock.json
```
Docker now runs `npm install` (not `npm ci`), generating a fresh lock file inside the container.

**Lesson**: Either pin npm version globally in your team, or exclude `package-lock.json` from Docker and let each environment manage its own. The second approach is simpler for solo/small teams.

---

### 3. Prisma Client Not Found at Runtime

**What broke**: Docker container started but crashed with "Cannot find module '.prisma/client'".

**Why it happened**: `npx prisma generate` was being run before `COPY . .` in the Dockerfile, so the schema wasn't available when generation ran.

**Fix**:
```dockerfile
# Wrong order:
RUN npx prisma generate
COPY . .

# Correct order:
COPY . .
RUN npx prisma generate
```

**Lesson**: Prisma's generated client is based on your `schema.prisma`. Always copy the source first, then generate.

---

### 4. Old SQLite Migrations Blocking PostgreSQL

**What broke**: `prisma migrate deploy` failed on startup because the `prisma/migrations/` folder contained old SQLite-format migrations that were incompatible with PostgreSQL.

**Why it happened**: The project originally used SQLite for simplicity, then switched to PostgreSQL for Docker. The migration history wasn't cleaned up.

**Fix**: Deleted the entire `prisma/migrations/` folder and ran `prisma migrate dev --name init` to generate a fresh PostgreSQL migration from the current schema.

**Lesson**: When switching databases, clear your migration history. Prisma migrations are database-specific. The schema itself is portable; the migration SQL is not.

### 5. Next.js 15 `params` is Now a Promise

**What broke**: Dynamic page components (`app/card/[linkToken]/page.tsx`, API routes) crashed with type errors or runtime failures when accessing `params.linkToken`.

**Why it happened**: Next.js 15 changed `params` from a plain object to `Promise<{...}>` in server components. Accessing it directly without `await` returns the Promise, not the value.

**Fix**:
```typescript
// Server components (pages):
export default async function CardPage({ params }: { params: Promise<{ linkToken: string }> }) {
  const { linkToken } = await params;  // ŌåÉ await required
}

// Client components (if needed):
import { use } from 'react';
const { linkToken } = use(params);
```

**Lesson**: When upgrading to Next.js 15, audit every file that uses `params` or `searchParams`. The TypeScript types don't always catch this immediately.

### 6. Stripe ŌåÆ Paddle Migration

**What changed**: The original payment provider was Stripe. We migrated to Paddle because it handles VAT/tax collection automatically (important for EU customers) and has simpler pricing for small platforms.

**What was involved**:
- Replaced `stripe` npm package with `@paddle/paddle-node-sdk`
- Changed env vars: `STRIPE_SECRET_KEY` ŌåÆ `PADDLE_API_KEY`, `STRIPE_WEBHOOK_SECRET` ŌåÆ `PADDLE_WEBHOOK_SECRET`
- Webhook event changed: `checkout.session.completed` ŌåÆ `transaction.completed`
- Paddle uses "Price IDs" (`pri_...`) instead of calculating amounts in code
- `docker-compose.yml` updated to use Paddle env vars

**What we didn't change**: The Prisma field `stripePaymentId` was kept (stores Paddle transaction ID now) to avoid running another DB migration. It's a cosmetic naming issue; the data is correct.

**Lesson**: When migrating payment providers, list every integration point: npm package, env vars, webhook events, checkout session creation API, webhook handler, and any field names in your schema. Plan the rename of schema fields up front to avoid technical debt.

### 7. Redis Caching for Link Validation

**Pattern established**:
1. On card creation: `SETEX card:{linkToken} 604800 {cardId, expiresAt}`
2. On card access: `GET card:{linkToken}` ŌĆö if missing, card is expired or invalid
3. Double-check with `expiresAt` timestamp in case Redis TTL drifts
4. Fallback to PostgreSQL if Redis is unreachable

**Why this order matters**: Redis can be evicted (maxmemory policy) before TTL expires. The double-check prevents false "valid" reads. The DB fallback means Redis being down doesn't break card viewing.

## Architectural Decisions Made

### Decision: HTML/CSS rendering over Canvas

Cards are rendered as React components with Tailwind CSS, not drawn on an HTML5 Canvas.

**Why**:
- Responsive across all screen sizes automatically
- Accessible (screen readers, keyboard navigation)
- SEO-friendly (OG image metadata can reference the content)
- React component model is easier to maintain
- Framer Motion integrates natively

The download-as-PNG feature uses `html2canvas` to snapshot the rendered HTML when needed.

### Decision: Service layer architecture

Business logic lives in `lib/services/` (cardService, linkService, emailService, paymentService, templateService), not in API route handlers.

**Why**:
- API routes stay thin (validate input ŌåÆ call service ŌåÆ return response)
- Services can be called from multiple routes (e.g., webhook and dev mock both call `sendCardEmailsToRecipients`)
- Easier to test in isolation
- Clear separation of concerns

### Decision: sessionStorage for multi-step form state

Card creation is a 3-step flow (customise ŌåÆ recipients ŌåÆ checkout). State is persisted in `sessionStorage` between steps.

**Why not URL params**: Too much data (full design config), not secure.
**Why not global state (Redux/Zustand)**: Overkill for a linear flow; sessionStorage survives page refreshes.
**Cleared after**: Payment confirmation.

### Decision: Webhook idempotency

The webhook handler checks `card.isPaid` before processing. If already `true`, it logs and returns 200 immediately.

**Why**: Payment providers retry webhooks on failure or timeout. Without idempotency, recipients would get duplicate emails on every retry.

### Decision: Email fallback to console

If `SENDGRID_API_KEY` is not set, the email service logs a rich console preview instead of sending.

**Why**: Makes local development easy (no SendGrid account needed), doesn't break the flow, and makes it obvious what would be sent in production.

## Things to Do Differently Next Time

1. **Pick the payment provider before writing code.** Migrating from Stripe to Paddle mid-build cost ~2 hours. Evaluate requirements (tax handling, geography, pricing model) upfront.

2. **Delete migration history when switching databases.** Don't let old SQLite migrations linger in a PostgreSQL project.

3. **Pin the npm version in Dockerfile.** `FROM node:20-alpine` gives npm@10; if your local machine runs npm@11, you'll have lock file conflicts. Use `RUN npm install -g npm@11` or use `--ignore-scripts` / `--omit=optional` defensively.

4. **Create one doc per phase, not one doc per milestone.** Having `PHASE_3_IN_PROGRESS.md`, `PHASE_3_COMPLETE.md`, `PHASE_3_TESTING.md`, `PHASE_3_QUICK_START.md`, and `README_PHASE_3.md` all at once for the same phase is overwhelming and they quickly go stale. One file, updated in place.

5. **Extract shared logic before it gets copied.** The `shareCard` and `ShareButton` duplication between ScrollToOpen and ClickToReveal was obvious from the start but wasn't extracted until today. Extract as soon as you write something the second time.
